# COG-5: Asset Metadata

```
Status:      Draft (Work in Progress)
Version:     0.1
Created:     2025-01-23
Authors:     Mike Anderson
```

:::caution Work in Progress
This specification is under active development. Structure and details may change significantly based on implementation experience and community feedback.
:::

This standard specifies the metadata format for assets on the Covia Grid, enabling interoperability, discoverability, and verification of resources across the distributed network.

## Purpose

Assets are the fundamental resources of the Covia Grid, representing data, operations, and other computational resources that can be shared and utilised across venues. A well-defined metadata format is essential for:

- **Discovery**: Enabling clients to find relevant assets based on descriptive information
- **Verification**: Ensuring content integrity through cryptographic hashes
- **Interoperability**: Allowing assets to be understood and processed by any Grid participant
- **Provenance**: Tracking the origin, authorship, and licensing of resources

This specification defines the JSON-based metadata format that describes assets on the Grid.

## Terminology

See [COG-1: Architecture](./COG-001) for definitions of Grid terminology including Asset, Artifact, Operation, and Job.

## Specification

### Metadata Format

Asset metadata MUST be a valid JSON object.

Asset metadata MUST be serialised as a UTF-8 encoded string for the purpose of computing the Asset ID.

Implementations SHOULD use canonical JSON formatting (sorted keys, no unnecessary whitespace) when computing Asset IDs for consistency, though this is not strictly required as long as the exact metadata string is preserved.

### Asset ID Computation

The Asset ID is computed as follows:

1. Serialise the metadata as a UTF-8 JSON string
2. Compute the SHA256 hash of the UTF-8 bytes
3. Encode the hash as a lowercase hexadecimal string (64 characters)

Example:
```
Metadata: {"name":"Example Asset","description":"A test asset"}
Asset ID: 7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b
```

Implementations MUST preserve the exact metadata string used to compute the Asset ID.

### Common Fields

The following fields are RECOMMENDED for all assets:

#### `name` (string)

A human-readable name for the asset.

```json
{
  "name": "Iris Dataset"
}
```

#### `description` (string)

A detailed description of the asset, its purpose, and how it can be used.

```json
{
  "description": "The famous Iris flower dataset for machine learning classification tasks"
}
```

#### `creator` (string)

The creator or author of the asset.

```json
{
  "creator": "UCI Machine Learning Repository"
}
```

#### `dateCreated` (string)

The creation date in ISO 8601 format.

```json
{
  "dateCreated": "2025-06-05T06:53:59Z"
}
```

#### `dateModified` (string)

The last modification date in ISO 8601 format.

```json
{
  "dateModified": "2025-06-05T07:22:59Z"
}
```

#### `keywords` (array of strings)

Keywords for discovery and categorisation.

```json
{
  "keywords": ["machine learning", "dataset", "classification"]
}
```

#### `license` (object)

Licensing information for the asset.

```json
{
  "license": {
    "name": "CC BY 4.0",
    "url": "https://creativecommons.org/licenses/by/4.0/"
  }
}
```

### Content Descriptor

Assets that include associated content (files, data) MUST include a `content` object with the following fields:

#### `content.sha256` (string, REQUIRED)

The SHA256 hash of the content, encoded as a hexadecimal string.

#### `content.contentType` (string)

The MIME type of the content.

#### `content.encoding` (string)

The character encoding for text content (e.g., "UTF-8").

#### `content.inLanguage` (string)

The language of the content using BCP 47 language tags.

Example artifact metadata with content:

```json
{
  "name": "Iris Dataset",
  "creator": "UCI Machine Learning Repository",
  "description": "Classic machine learning dataset for classification",
  "dateCreated": "2025-06-05T06:53:59Z",
  "license": {
    "name": "CC BY 4.0",
    "url": "https://creativecommons.org/licenses/by/4.0/"
  },
  "keywords": ["machine learning", "iris", "dataset"],
  "content": {
    "contentType": "text/csv",
    "sha256": "119E30DB8A4EA8B33723603743591A5F8229684E6236D89EF1966A72D7293607",
    "encoding": "UTF-8",
    "inLanguage": "en-GB"
  }
}
```

### Operation Descriptor

Assets that represent executable operations MUST include an `operation` object with the following structure:

#### `operation.adapter` (string)

The adapter responsible for executing the operation. This determines how the operation is processed.

Common adapter values include:
- `http` - HTTP request operations
- `jvm` - JVM-based operations
- `orchestrator` - Multi-step workflow orchestration
- `mcp` - Model Context Protocol operations
- `langchain` - LangChain-based AI operations
- `convex` - Convex blockchain operations
- `grid` - Grid meta-operations

#### `operation.input` (object)

JSON Schema defining the expected input format.

```json
{
  "operation": {
    "input": {
      "type": "object",
      "properties": {
        "url": {
          "type": "string",
          "description": "The URL to fetch"
        }
      },
      "required": ["url"]
    }
  }
}
```

#### `operation.output` (object)

JSON Schema defining the expected output format.

```json
{
  "operation": {
    "output": {
      "type": "object",
      "properties": {
        "status": {
          "type": "number",
          "description": "HTTP status code"
        },
        "body": {
          "type": "string",
          "description": "Response body"
        }
      }
    }
  }
}
```

#### `operation.info` (object)

Additional metadata about the operation.

```json
{
  "operation": {
    "info": {
      "featured": true,
      "category": "utilities"
    }
  }
}
```

### Orchestration Operations

Operations that orchestrate multiple sub-operations include additional fields:

#### `operation.steps` (array)

An ordered array of steps to execute.

Each step object contains:
- `op` (string): The operation identifier (adapter:operation or asset ID)
- `name` (string): Human-readable step name
- `input` (object|array): Input mapping for the step

#### `operation.result` (object)

Mapping that defines how to construct the final output from step results.

Example orchestration:

```json
{
  "name": "Search and Summarise",
  "operation": {
    "adapter": "orchestrator",
    "input": {
      "type": "object",
      "properties": {
        "query": {
          "type": "string",
          "description": "Search query"
        }
      },
      "required": ["query"]
    },
    "steps": [
      {
        "op": "http:get",
        "name": "Search",
        "input": {
          "url": ["input", "query"]
        }
      },
      {
        "op": "langchain:summarise",
        "name": "Summarise Results",
        "input": [0, "body"]
      }
    ],
    "result": {
      "summary": [1, "text"],
      "searchResults": [0]
    }
  }
}
```

### Input Mapping

Step inputs in orchestrations use a path-based mapping syntax:

| Syntax | Description |
| ------ | ----------- |
| `["input"]` | The entire input to the orchestration |
| `["input", "field"]` | A specific field from the input |
| `[0]` | Output of step 0 |
| `[1, "field"]` | A specific field from step 1's output |
| `["const", value]` | A constant literal value |

### Additional Information

The `additionalInformation` object MAY contain any implementation-specific or domain-specific metadata:

```json
{
  "additionalInformation": {
    "notes": ["Uploaded for testing purposes"],
    "sourceUrl": "https://example.com/original"
  }
}
```

## Schema Compatibility

Input and output schemas SHOULD follow JSON Schema conventions for maximum interoperability.

Implementations MAY support extended type values beyond standard JSON Schema types:

| Type | Description |
| ---- | ----------- |
| `asset` | Reference to another asset on the Grid |
| `any` | Any valid JSON value |

## Validation

Implementations SHOULD validate metadata against this specification before accepting assets.

Implementations MUST reject metadata that:
- Is not valid JSON
- Contains content hashes that do not match the actual content
- References non-existent adapters (when executing operations)

## Security Considerations

### Content Verification

The `content.sha256` field enables verification of content integrity. Implementations MUST verify content hashes before trusting content data.

### Metadata Immutability

Once an asset is created, its metadata string and Asset ID are immutable. Any modification to metadata results in a new Asset ID. Implementations MUST NOT allow modification of existing metadata.

### Input Validation

Operations SHOULD validate inputs against the declared `operation.input` schema before execution to prevent injection attacks or malformed data.

## Examples

### Artifact (Dataset)

```json
{
  "name": "Iris Dataset",
  "creator": "UCI Machine Learning Repository",
  "description": "Classic flower dataset for classification",
  "dateCreated": "2025-06-05T06:53:59Z",
  "dateModified": "2025-06-05T06:53:59Z",
  "license": {
    "name": "CC BY 4.0",
    "url": "https://creativecommons.org/licenses/by/4.0/"
  },
  "keywords": ["machine learning", "iris", "dataset", "classification"],
  "content": {
    "contentType": "text/csv",
    "sha256": "119E30DB8A4EA8B33723603743591A5F8229684E6236D89EF1966A72D7293607",
    "encoding": "UTF-8",
    "inLanguage": "en-GB"
  }
}
```

### Operation (HTTP GET)

```json
{
  "name": "HTTP GET Operation",
  "description": "Performs a simple HTTP GET request",
  "dateCreated": "2025-01-27T10:00:00Z",
  "operation": {
    "adapter": "http",
    "input": {
      "type": "object",
      "properties": {
        "url": {
          "type": "string",
          "description": "The URL to request"
        },
        "headers": {
          "type": "object",
          "description": "Optional HTTP headers"
        }
      },
      "required": ["url"]
    },
    "output": {
      "type": "object",
      "properties": {
        "status": {
          "type": "number",
          "description": "HTTP status code"
        },
        "body": {
          "type": "string",
          "description": "Response body"
        }
      }
    }
  }
}
```
